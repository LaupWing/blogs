# De onzichtbare tijdbom in jouw website of software

Sinds de opkomst van AI lijkt het alsof iedereen een app kan bouwen. En ik wil hier één ding duidelijk maken: het *lijkt* zo.

De media schildert AI-tools af als het einde van software developers. Dat klopt niet. Wat wél klopt: de meeste mensen hebben geen idee van softwareontwikkeling maar kunnen nu ineens een minimale app in elkaar draaien.

Dat creëert een illusie van kundigheid. En die illusie kan je snel geld kosten — omdat je denkt dat iemand verstand heeft van iets wat hij achteraf niet heeft. Dat soort mensen noem ik kwakzalvers: ze leveren iets op dat eruitziet als een product, maar de fundering ontbreekt volledig.

Ik heb 10 jaar softwareontwikkeling ervaring en gebruik dagelijks AI in mijn werk. Ik werkte als lead developer en heb developers onder mij gehad die zwaar leunden op AI. Ik heb de fouten van dichtbij gezien.

In dit artikel leg ik uit waarom je mogelijk geïnvesteerd hebt in een website of software die je uiteindelijk niet meer kunt gebruiken.

## Wat architectuur is (zonder jargon)

Wat kan AI heel goed? Code schrijven. De details, de syntax, de standaard patronen — daar blinkt het in uit. Wat kan AI minder goed? De architectuur van je software bepalen.

Je denkt waarschijnlijk: wat is architectuur in software eigenlijk? Maak je geen zorgen — zelfs veel developers weten het niet goed te uitleggen.

Het beste voorbeeld: de fundering en het plattegrond van een huis. Voordat je gaat bouwen, stippel je alles uit. Klopt de indeling? Past alles? Is de fundering sterk genoeg? Als die fundering niet klopt, kan het hele huis instorten.

Zo werkt het ook met software. Vanaf dag één moet je uitdenken welke technologieën je gebruikt, hoe je mappenstructuur eruitziet, welke databases van toepassing zijn, wat beveiligd op de server moet leven — enzovoorts. Dit zorgt er niet alleen voor dat je software goed draait, maar ook dat het makkelijk uit te breiden is zonder alles opnieuw te moeten schrijven.

Je bent niet bezig om steentjes te leggen. Je tekent de map van de software. Steentjes leggen kan AI goed. De map tekenen niet.

Waarom weten zelfs de meeste developers dit niet? Simpel: architectuur vereist jaren ervaring. Er liggen zoveel lagen achter, en één laag zoals de database heeft al honderden sublagen. Die kennis moet voor elk specifiek project worden toegepast, want elk project heeft zijn eigen specificaties — en ook die specificaties goed uitstippelen vereist ervaring. Dat heeft AI niet. Een junior developer ook niet. En veel medior developers ook niet.

## Hoe de tijdbom tikt

Nu je weet wat architectuur is, schets ik een scenario.

Stel: je wilt een custom applicatie en huurt iemand in die "lijkt" te weten wat hij doet — want hij heeft een simpele app gebouwd met AI.

Alles lijkt te werken. Hier en daar wat foutjes, maar het werkt voor 80%. Die mirage houdt het hooguit een paar maanden vol. Dan begin je de echte fouten te zien: verkeerde onderdelen worden bijgewerkt, data mist op verschillende plekken, afbeeldingen laden niet. Dit zijn kleine gaten die de slechte fundering verraden.

Je gaat terug naar de "developer" en vraagt om fixes. Hier heb ik slecht nieuws: die laatste 20% kost waarschijnlijk meer dan 80% van het echte werk. De laatste fouten eruit halen duurt meerdere keren langer dan alles wat daarvoor is gebouwd.

Waarom? Omdat degene die het gebouwd heeft geen idee heeft wat hij gedaan heeft. Alles is uit de losse pols geschreven. De hele codebase moet worden doorgenomen, geherstructureerd — voor zover dat überhaupt mogelijk is.

Ik kan hier een concreet voorbeeld van geven. Ongeveer 5-6 jaar geleden begon ik aan een webapplicatie voor een klant: een platform voor juridische drone documentatie, zodat regelgeving makkelijk bijgehouden en geüpdatet kon worden. Nuchter ontwerp, functioneel — geen fancy interface. Maar de data onder de motorkap was dat ook niet. Ik koos voor een simpele database die weinig structuur nodig had, zodat ik snel de voorkant kon bouwen.

Die ene keuze bijt mij nog steeds. De data is door de jaren heen zo onleesbaar geworden dat het door de hele applicatie trekt — hoewel de voorkant goed werkt, is de data aan de achterkant heel losjes. Gelukkig heb ik het ondertussen rechtgetrokken, maar dit kostte mij maanden werk.

Zorg dat de fundering goed staat. Want als het ontploft, ontploft niet alleen je app — ook je bankrekening.

## Waarom AI dit erger maakt

Het probleem gaat met een sneltreinvaart de verkeerde kant op.

Goede developers waarderen simpele, goed opgebouwde code. Slechte developers beoordelen hun werk op de hoeveelheid code die ze schrijven.

Goede developers sturen AI om sneller goede funderingen te leggen. Slechte developers laten AI op topsnelheid slechte code schrijven.

Goede fundering zorgt voor snellere ontwikkeling later en exponentieel betere resultaten. Slechte fundering zorgt voor exponentieel grotere problemen, sneller.

Dat is de wereld waar we nu in leven. Mensen die denken dat ze software kunnen bouwen terwijl ze nauwelijks weten hoe ze hun laptop aanzetten. Dit frustreert mij enorm — ik heb jaren moeten ploegen om goede, werkende software te bouwen. Ja, dit is persoonlijk. Maar het raakt jou ook: jij bent degene die straks veel geld moet uitgeven aan iemand die er niets van begrijpt.

## Hoe je het herkent voordat je tekent

Hoe voorkom je dit? Het is helaas een lastig probleem: zonder verstand van software, hoe zie je de signalen?

De beste concrete vragen die je kunt stellen:

**Hoe lang ontwikkelt deze persoon al software?** Niet hoelang ze "bezig zijn met AI" — maar echt programmeren.

**Vraag naar hun GitHub.** Hoe lang is het zichtbaar dat ze programmeren? Is het recent aangemaakt of loopt het al jaren terug?

**Kunnen ze werkende producten laten zien?** Hoe ziet dat product eruit? Zoek op ChatGPT naar simpele softwareparadigma's en check of de persoon het correct toepast.

Om een voorbeeld van mezelf te geven: ik heb meer dan 500 projecten op mijn GitHub staan. Tientallen webapplicaties die live staan op het internet. Taloze klantprojecten. Dit zijn meetbare, visuele voorbeelden — daar heb je weinig technische kennis voor nodig om het te zien.

## Bouw het goed, of betaal het twee keer

Als je wilt bouwen, zorg dat je iemand inhuurt die de fundering serieus neemt — niet iemand die snel 80% in elkaar zet en hoopt dat de rest volgt.

Achteraf repareren kost altijd meer dan het goed doen. Niet een beetje meer. Veel meer.
