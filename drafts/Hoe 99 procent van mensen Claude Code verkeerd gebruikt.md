# Hoe 99% van mensen Claude Code verkeerd gebruikt

## Intro
- iedereen gebruikt Claude Code, weinigen gebruiken het goed
- het verschil zit niet in de tool maar in wie de tool gebruikt
- 10 jaar coderen, 500+ projecten, dagelijks programmeren — ik zie het verschil

## 1: Ze gebruiken het om snel iets te maken, niet om vragen te stellen
- Claude Code als shortcut, niet als sparringpartner
- de beste output komt van de beste input — garbage in, garbage out
- goede developers stellen eerst de vraag, dan pas bouwen

## 2: Ze beoordelen lines of code in plaats van kwaliteit van code
- meer code = meer werk = betere developer — dat klopt niet
- AI genereert makkelijk 500 regels waar 50 genoeg is
- als je niet weet wat goede code is, zie je het verschil niet

## 3: Geen fundamentele kennis maar wel iets kunnen maken — de Dunning-Kruger illusie
- AI maakt het makkelijk om iets neer te zetten zonder te begrijpen wat je neerzet
- je kan een huis bouwen zonder te weten hoe fundamenten werken — totdat het instort
- gevaarlijkste type: genoeg kennis om iets te maken, te weinig om te zien wat er mis is

## 4: Ze redirecten nooit
- persoonlijk verhaal: talloze keren de coding agent moeten bijsturen
- concreet voorbeeld: dubbele property keys op data object — als ik het niet gecheckt had stond het er gewoon in
- AI schrijft wat je vraagt, niet wat je bedoelt — jij bent de architect, niet de AI
- zonder fundamentele kennis zie je de fouten niet eens

## 5: Systems thinking is alleen mogelijk als je echte systems gebouwd hebt
- je kan niet over architectuur nadenken als je nog nooit een systeem hebt onderhouden
- concreet: Dronestart (drone document consultancy app) 4 keer opnieuw gebouwd — elke keer betere architectuur, betere keuzes, minder technische schuld
- de eerste versie werkte. de vierde versie schaalt. dat verschil leer je niet in een cursus
- AI geeft je het eindresultaat zonder het proces — het proces is waar je het leert
- voor een ondernemer: iemand die een systeem 4 keer heeft herbouwd begrijpt wat er mis gaat. iemand die het met AI in een week heeft neergezet niet

## 6: Ze geven geen context mee
- Claude Code weet niks over jouw project tenzij je het vertelt
- mensen typen "fix this bug" zonder uitleg van wat het systeem doet, waarom de code zo staat, wat de constraints zijn
- resultaat: een oplossing die werkt maar niet past — en over 3 maanden een nieuw probleem

## 7: Ze accepteren de eerste output zonder te reviewen
- AI is zelfverzekerd, ook als het fout zit
- een goede developer reviewt elke regel alsof een junior het schreef
- zonder fundamentele kennis kan je niet reviewen — je copy-pasted blind en hoopt dat het klopt

## Conclusie
- Claude Code is een multiplier — het maakt goede developers beter en slechte developers gevaarlijker
- de tool is niet het probleem, de gebruiker is het probleem
- ken je fundamenten, stel de juiste vragen, redirect waar nodig
